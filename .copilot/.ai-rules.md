## AI Project Rules & Standards
### 1. Core Technical Stack & Environment
Runtime: Python 3.12.2+ (Strict).

Dependency Management: Use pyproject.toml with uv or poetry. No requirements.txt.

Type Checking: Use pyright or mypy in strict mode.

Modern IO: Use pathlib for all file system operations. Avoid os.path.

### 2. Python 3.12+ Coding Standards
Generics (PEP 695): Use the new type statement and square bracket syntax for generics.

Preferred: def handle_data[T](item: T) -> T:

Avoid: T = TypeVar("T")

Pattern Matching: Use match-case for complex branching and structural validation.

Type Aliases: Use the type keyword (e.g., type UserID = int) instead of typing.TypeAlias.

F-Strings: Utilize 3.12’s lifted f-string constraints (nested quotes and multiline expressions are allowed).

### 3. Architectural Constraints & Async
Concurrency: Always use asyncio.TaskGroup() for managing concurrent tasks to ensure structured concurrency.

Error Groups: Use except* when handling ExceptionGroup in asynchronous blocks.

State: Prefer @dataclass(slots=True, frozen=True) for immutable data structures to optimize memory and prevent side effects.

Functional Patterns: Use list/dict comprehensions for simple transformations; use itertools for memory-efficient complex iterations.

### 4. Security & Error Handling
Secrets: Never hardcode credentials. Use python-dotenv or environment variables exclusively.

Validation: All external data (API/File) must be parsed via Pydantic v2 models.

Explicit Exceptions: Never use bare except:. Always catch specific exceptions and log them with context.

SQL Safety: Use parameterized queries for all SQL. Prefer lightweight DB access (e.g. raw `sqlite3` for CLI tools) or SQLAlchemy 2.0 for larger applications.

### 5. Test Coverage & Documentation
Framework: Use pytest with pytest-asyncio.

Requirement: Every new feature must include a corresponding test in the tests/ directory.

Mocking: Use unittest.mock or pytest-mock to isolate external API dependencies.

Docstrings: Use Google Style docstrings. Focus on the "Why" and "Edge Cases" rather than describing obvious code logic.

### 6. AI Communication & Implementation Style
Directness: Skip conversational filler. Provide code solutions first, followed by a concise "Technical Reasoning" section.

Diff-Only: When modifying existing files, provide only the updated functions or lines unless a full rewrite is requested.

Self-Correction: Before outputting code, verify it against Python 3.12 syntax (especially the new Type Parameter syntax).

Tool Usage: During the "Planning" phase, always check the existing directory structure before suggesting new file paths.

### 7. The "Anti-Pattern" List (Never Do These)
No asyncio.gather(): Use TaskGroup instead.

No typing.List or typing.Dict: Use native list[] and dict[] types.

No global keyword: Use dependency injection or class-based state.

No pkg_resources: Use importlib.metadata.

### 8. Implementation stepping process
When working off a .tasks.md file:
1. Mark the current step in progress in the .tasks.md file
2. Read related test definitions from related .tests.md file
3. Implement tests and ensure they are failing
4. Implement functionality and iterate until all tests pass
5. Run type checkers and linters repeatedly fixing all errors and warnings between runs until no warnings or errors are presented
6. Mark the current step completed in the .tasks.md file
